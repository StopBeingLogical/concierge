# Concierge: Technical Specification

*An implementation reference document — written to be read when building begins. The philosophy doc tells you what Concierge is and why. This document tells you what Concierge does and how the pieces fit together. Schemas before code. Contracts before implementation.*

*v5 — March 2026*

---

## Section 1: Layer Interface Contracts

Concierge operates as a five-layer stack. Each layer has exactly one job. Each layer communicates only with its immediate neighbors. No layer has knowledge of non-adjacent layers. This is not a guideline — it is the invariant that makes the system legible, testable, and extensible.

The layers in order:

```
Bit → Planner → Router → Foreman → Workbee
```

Every artifact passing between layers is a typed, versioned, schema-validated object. No freeform dictionaries. No implicit contracts. If it crosses a layer boundary it has a schema.

---

### Layer 1 — Bit

**Job:** The human-facing application. Captures intent. Translates it into a deterministic artifact. Delivers results back to the human. Acts as a first-level reasoning and conversational agent when the Concierge stack is unavailable or inappropriate. The human interacts only with Bit and never needs to know or care what operates below it.

**Communicates with:** Planner only. All stack interaction is mediated through this boundary.

**Emits:** Intent Artifacts, Approval Signals, Queue Flush Requests

**Receives:** Execution Plans (for human approval), Result Summaries, Status Events, Gap Reports, Capability Availability Responses

#### Bit Operating Modes

**Connected mode:** Bit delegates to the Concierge stack. Intent Artifacts are passed to the Planner. Results, status, and gap reports return through the same channel. This is the preferred operating mode.

**Disconnected mode:** Bit operates on its own resources — local LLM inference, local filesystem, local corpus cache, web search and fetch from approved sources. Bit can conduct conversational and reasoning tasks in this mode. Intent Artifacts that require the Concierge stack are queued locally rather than dropped.

**Reconnection handshake:** When the Planner connection is reestablished, Bit surfaces any queued Intent Artifacts to the human for review before flushing to the Planner. The human confirms, modifies, or discards each queued item. Nothing flushes automatically.

#### Bit Local Resources

**Local LLM inference** — sufficient for conversational tasks, intent drafting, light reasoning, and rubber-duck dialogue. Not a replacement for the ensemble — a fallback. Target backend on Apple Silicon: Apple Neural Engine via CoreML/MLX. Current implementation: Ollama (localhost:11434) or llama.cpp (localhost:8000), default model qwen2.5:14b.

**Local corpus cache** — the most recent background prefill output generated by the prefill Task Package. Represents the system's current best pre-positioned context across active domains. Kept current by the prefill Task Package running at P2/P3 when the stack is available.

**Local filesystem access** — Bit has direct read/write access to the local filesystem for artifact delivery, document rendering, and cache management. This is not a violation of the Lock List — local filesystem access is an explicit Bit capability, not an execution capability.

**Web search and fetch** — available in both connected and disconnected modes. Bit may consult approved external sources to supplement local knowledge. Web-sourced content is clearly distinguished from corpus-sourced content in all outputs.

#### Bit Application Surface

Bit is a full-featured application. The following are application-layer capabilities, not layer contract concerns:

- Media viewers and document renderers
- Gallery viewer and processing modals for artifact review
- System dashboards and status surfaces — including the `Harkanza` queue (denied jobs awaiting human revision and resubmission), the capability gap project queue, and the Hašatar task queue
- Control modals for queue management, node status, task package review
- Notification and alert surfaces

The detailed Bit application specification is a separate document. This spec concerns only Bit's role as Layer 1 in the Concierge stack and the contracts it maintains with Layer 2.

#### Bit Hard Boundaries — The Lock List

Bit does not execute work chunks. Bit does not call models directly on behalf of the stack. Bit does not communicate with Router, Foreman, or Workbee. Bit does not approve its own jobs. Bit does not modify an Intent Artifact after emission.

Any capability not on the explicit allowed list is forbidden by default. The Lock List is enforced architecturally — violations are architecture errors, not implementation bugs.

**`Harkanza` holding state:** when a human denies an Execution Plan, the job moves to `Harkanza` — a named holding state visible in Bit's dashboard. The job is not discarded and does not become `failed`. It is available for revision: the human may edit the intent, request a new plan from the Planner, and resubmit. The human may also cancel a `Harkanza` job explicitly. Nothing in `Harkanza` is acted upon automatically.

#### Intent Artifact Schema

```json
{
  "schema_version": "1.0",
  "intent_id": "<uuid v5 — deterministic from SHA256 of canonical content>",
  "issued_at": "<ISO8601>",
  "sha256": "<hash of canonical intent fields>",
  "source": "bit",
  "raw_input": "<original human expression>",
  "connection_state": "<connected|disconnected>",
  "queued_at": "<ISO8601 — null if not queued>",
  "mode": "<chat|code|snap|xform — audit only, does not affect schema shape>",
  "intents": [
    {
      "rank": 1,
      "intent": "<candidate interpretation in normalized natural language>",
      "confidence": 0.0,
      "evidence": ["<fragment from raw_input supporting this interpretation>"],
      "required_inputs": ["<input identifiers needed to execute this interpretation>"],
      "inferred_constraints": ["<constraints extracted from raw_input>"],
      "ambiguity_flags": ["<aspects of this interpretation that are uncertain>"]
    }
  ],
  "questions": [
    {
      "question_id": "<uuid>",
      "question": "<clarification question text>",
      "blocking": true,
      "reason": "<why this question affects interpretation>",
      "default_assumption": "<null if blocking — stated assumption if non-blocking>"
    }
  ],
  "status": "<ready|needs-clarification>",
  "requires_approval": true
}
```

**Invariants:**
- `intent_id` is UUID v5 derived from SHA256 of canonical content — same substantive input always produces the same ID
- `job_id` is UUID v4 (random) — each job is unique even from the same intent
- `mode` is captured for audit traceability only — must not influence schema shape, required fields, or execution behavior
- `status: ready` means top-ranked intent has sufficient confidence and no blocking questions remain — Planner may proceed to plan generation
- `status: needs-clarification` means blocking questions exist — Planner surfaces them to Bit before proceeding
- Non-blocking questions carry a `default_assumption` — Planner proceeds with stated assumption if human does not answer, and logs the assumption in the execution record
- Bit never modifies an Intent Artifact after emission
- `queued_at` is set at queue time and preserved through flush

#### Disambiguation UI Behavior

When the Planner returns multiple intent candidates, Bit presents them using the following pattern:

- Top-ranked candidate is pre-selected as the default with its confidence score visible
- A disclosure control (dropdown or modal callout) reveals the full ranked list, per-candidate evidence, and clarification questions on demand
- Blocking questions surface inline and must be answered before the confirm action is available
- Non-blocking questions display the Planner's stated default assumption — the human may override or accept
- Confirmation is always required regardless of confidence level — high confidence makes the step fast, not skippable

### Layer 2 — Planner

**Job:** Receives Intent Artifacts from Bit. Matches against the Task Package Registry above a confidence threshold. Queries the Router for tool availability before committing to a plan. Generates an Execution Plan. Returns the plan to Bit for human approval before passing anything to the Router. Validates results on the return path. Owns the Gap Resolver and the Task Package Creation Wizard.

**Communicates with:** Bit (receives intent, returns plans and gap reports), Router (submits approved jobs, queries tool availability).

**Transport:** HTTP JSON over localhost or LAN. Bit runs a local HTTP client. The Workbee stack exposes eight endpoints: `GET /health`, `POST /jobs`, `POST /jobs/{id}/plan`, `POST /jobs/{id}/runs`, `GET /jobs/{id}/runs/{run_id}`, `GET /jobs/{id}/runs/{run_id}/events` (SSE), `GET /jobs/{id}/runs/{run_id}/logs` (JSONL), `GET /jobs/{id}/runs/{run_id}/artifacts`. WorkbeeSim implements the same API surface for testing against deterministic fixture scenarios.

**Emits:** Execution Plans, Job Specs, Gap Reports, Tool Availability Queries, Task Package Creation requests

**Receives:** Intent Artifacts, Approval Signals, Work Results (from Router), Tool Availability Responses

#### Planner Intent Matching Behavior

**Step 1 — Registry match.** The Planner queries the Task Package Registry for matching packages. If a match is found above the confidence threshold, proceed to Step 2. If below threshold, surface candidates to Bit and ask the human to elaborate or confirm. If no viable candidates exist, offer the Task Package Creation Wizard or Tarhuili mode.

**Step 2 — Tool availability query.** Before generating an Execution Plan, the Planner queries the Router: are the tools required to execute this package currently available in the tool pool? Three outcomes:

- **All tools available:** proceed to plan generation
- **Tools partially available:** surface a gap report to Bit. Offer to queue a Hašatar task alongside the original request. Original request status becomes `Handanza`.
- **No tools available:** same as partial. Hašatar task is queued at P2/P3 pending human confirmation. Original request waits.

The Gap Resolver is only invoked after the tool availability query confirms a gap. The Planner does not invoke the Gap Resolver speculatively.

**Step 3 — Plan generation.** Planner generates an Execution Plan from the matched Task Package. The plan enumerates every step, its compute requirements, its dependencies, and its failure handling. This plan is returned to Bit for human approval before any execution begins.

**Step 4 — Human approval.** No Job Spec is passed to the Router until an explicit approval signal is received from Bit. Task Packages carry pre-acknowledgement for their described action class — but execution of a specific package instance still requires confirmation. This step is invariant. It cannot be bypassed.

In Tarhuili mode the Planner enumerates its intended pipeline in the same Execution Plan format and presents it to the human for confirmation. The confirmation step is identical — the only difference is the plan was approximated rather than drawn from a registered package.

#### Foundation Generation

When a request cannot be serviced because required tools do not exist, the system does not fail. It defers and builds.

The Planner:
1. Tags the original request `Handanza`
2. Surfaces a gap report to Bit describing what is missing
3. Proposes a Hašatar task — a Tool Creation Task Package that builds the missing tools
4. Awaits human confirmation before queuing the foundation task
5. Routes confirmed foundation work through P2/P3 queue
6. On foundation completion, re-evaluates the original request and surfaces it to the human for re-confirmation before execution

Foundation generation tasks are themselves Task Packages. The system uses its own machinery to extend itself, within human-confirmed bounds.

#### Result Validation

On the return path, the Planner performs final validation of Work Results received from the Router:

- Verifies result integrity against the originating Job Spec SHA256 chain
- Evaluates outputs against the Task Package's acceptance criteria
- Accepts or declines
- Logs the outcome with full provenance
- On acceptance: passes result summary up to Bit
- On decline: passes failure report down to Router with reason, or up to Bit if the failure requires human decision

#### Gap Resolver

The Gap Resolver is a named Planner capability — not a separate layer or service.

Two gap types:

**Knowledge gap** — the Planner cannot match an intent because the relevant domain is absent or underrepresented in the corpus. Resolution: trigger a corpus enrichment task from approved sources. Surface the gap to Bit. Queue the original request pending enrichment.

**Capability gap** — the Planner cannot service a request because required tools do not exist in the tool pool. Resolution: foundation generation process as described above.

The Gap Resolver only activates after the tool availability query confirms the gap is real. The Planner does not hallucinate gaps.

#### Task Package Registry

Each Task Package specifies: input requirements and schema, execution pipeline, output format and return schema, validation and acceptance criteria, pre-acknowledgement scope, confidence matching rules.

The registry is versioned. Packages are immutable once registered. A new version supersedes the old — the old version remains available for in-flight jobs.

The Task Package Creation Wizard is a Planner-led process that produces a new registered Task Package. It requires human confirmation before the package is registered.

**Phase 2 fields:** `mounts`, `tools_allowed`, `model_policy`, `budget_policy`, and `telemetry_policy` are defined in the schema now and required for full implementation. **Invariants:**
- `tool_availability_verified` must be true before a plan is presented for approval — the Planner never presents a plan it hasn't verified can be executed
- `approximated: true` flags Tarhuili mode plans — the human sees this
- Planner does not modify a plan after presentation to Bit
- No step may declare a dependency on a step that appears later in the list
- `compute_class` is a requirement hint to the Router — not a hardware identity

### Layer 3 — Router

**Job:** Receives approved Job Specs from the Planner. Responds to tool availability queries. Decomposes execution plans into Work Chunks. Maintains the work pool. Aggregates results. Returns completed execution to the Planner.

**Design note:** The Router is the operational core of the Concierge system. Its internal scheduling logic is the subject of a dedicated design session with multi-model peer review. What is specified here is the Router's contract boundary only — what it accepts, what it emits, and what invariants it maintains at the layer interface. The internal implementation is explicitly deferred.

**Critical constraint:** The Router has zero knowledge of hardware identity. It sees capability envelopes, not node names, chip models, or vendor identities.

#### Tool Availability Response Schema

```json
{
  "schema_version": "1.0",
  "query_id": "<uuid>",
  "plan_id": "<uuid>",
  "queried_at": "<ISO8601>",
  "tools_required": ["<tool identifier>"],
  "availability": [
    {
      "tool_id": "<tool identifier>",
      "status": "<available|unavailable|degraded>",
      "node_count": 0,
      "notes": "<optional>"
    }
  ],
  "overall_status": "<ready|partial|blocked>"
}
```

**Invariants:**
- The Router responds to tool availability queries before any job is scheduled
- `overall_status: blocked` means no execution should proceed — foundation generation required
- `overall_status: partial` means degraded execution is possible — Planner surfaces this to human
- The Router does not infer tool availability — it reports only what the current node registry confirms

#### Work Chunk Schema

```json
{
  "schema_version": "1.0",
  "chunk_id": "<uuid>",
  "job_id": "<uuid>",
  "plan_id": "<uuid>",
  "step_id": "<uuid>",
  "issued_at": "<ISO8601>",
  "sha256": "<hash of chunk fields>",
  "action": "<worker action identifier>",
  "inputs": {},
  "compute_requirements": {
    "compute_class": "<cpu|igpu|dgpu|npu|any>",
    "memory_requirement_gb": 0,
    "accelerator_features": [],
    "max_latency_ms": 0
  },
  "timeout_ms": 0,
  "on_failure": "<abort|retry|skip>",
  "lease_ttl_ms": 0
}
```

**Invariants:**
- No chunk may name a specific node as its target
- `lease_ttl_ms` defines how long a node may hold a chunk before it is returned to the pool
- Router never assumes a chunk completed successfully until a verified Work Result is received

### Layer 4 — Foreman

**Job:** System-local coordinator. Owns heartbeat and capability advertisement for all nodes within its system. Pulls Work Chunks from the Router work pool. Validates chunk requirements against local node capability. Dispatches to the local Workbee. Returns Work Results to the Router.

**Communicates with:** Router (pulls chunks, returns results, sends capability advertisements and heartbeats), Workbee (dispatches execution units).

**Emits:** Work Results, Capability Advertisements, Heartbeats

**Receives:** Work Chunks

**Async buffer responsibility:** The Foreman maintains a local async buffer that absorbs RDMA jitter and load variance without blocking the Router or the verifier tier. Jitter affects diversity magnitude, not correctness.

#### Work Result Schema

```json
{
  "schema_version": "1.0",
  "result_id": "<uuid>",
  "chunk_id": "<uuid>",
  "job_id": "<uuid>",
  "step_id": "<uuid>",
  "node_id": "<uuid — opaque, assigned at registration, encodes no hardware identity>",
  "system_id": "<uuid — opaque system identifier>",
  "completed_at": "<ISO8601>",
  "sha256": "<hash of result fields>",
  "status": "<completed|failed|timeout|partial>",
  "output": {},
  "error": null,
  "error_details": null,
  "duration_ms": 0,
  "retries_used": 0
}
```

**Invariants:**
- `node_id` and `system_id` are opaque — they encode no hardware identity
- Result `sha256` covers `result_id`, `chunk_id`, `job_id`, `step_id`, `status`, `output`
- A failed result is a valid result — the Router and Planner handle failure routing
- Foreman never returns a result for a chunk it did not receive

---

### Layer 5 — Workbee

**Job:** Executes the work. Calls the tools. Uses whatever backend the hardware provides. Returns output to the Foreman. Has no knowledge of jobs, plans, chunks, or any layer above the Foreman.

**Communicates with:** Foreman only.

**Backend irrelevance:** Whether the Workbee uses CUDA, MLX, ROCm, OpenVINO, RKLLM, or CPU inference is irrelevant to every layer above it. The Foreman issued a contract. The Workbee fulfills it.

**Invariants:**
- Workbee does not communicate with any layer above Foreman
- Workbee does not maintain state between execution units
- Workbee does not make routing decisions
- If the Workbee cannot fulfill a contract, it returns an error — it does not attempt to route around it

### SHA256 Integrity Chain

Every artifact in the pipeline carries a SHA256 hash of its canonical fields, computed at emission and verified at receipt. The chain is:

```
Intent Artifact (sha256_1)
  → Execution Plan (sha256_2, covers sha256_1)
    → Approved Job Spec (sha256_3, covers sha256_2)
      → Work Chunk (sha256_4, covers sha256_3)
        → Work Result (sha256_5, covers sha256_4)
```

Each layer verifies the incoming hash before processing. A hash mismatch is a hard failure — the artifact is rejected and the job is marked failed with reason `integrity_violation`. No silent corruption propagates through the stack.

## Section 2: Job State Machine

A job is not a function call. It is a state machine with an explicit lifecycle, defined transitions, legible failure modes, and a full event log. Every state transition emits an event. Every event is logged. Nothing happens silently.

---

### Job States

| State | Meaning | Terminal |
|---|---|---|
| `draft` | Intent Artifact received by Planner, matching in progress | No |
| `pending_approval` | Execution Plan generated, awaiting human confirmation through Bit | No |
| `Harkanza` | Plan denied by human — job held for review and resubmission | No |
| `Handanza` | Plan requires tools that do not exist — Hašatar queued | No |
| `approved` | Human confirmed — Job Spec passed to Router | No |
| `queued` | Router has received the Job Spec, work not yet started | No |
| `running` | One or more chunks executing | No |
| `paused` | Execution suspended — awaiting human input (NEEDS_INFO) | No |
| `completed` | All steps finished successfully | Yes |
| `partial` | Execution finished with some steps failed, some succeeded — results available | Yes |
| `failed` | Execution failed — no usable results | Yes |
| `capability_exceeded` | Cannot service at acceptable quality — conditionally terminal | Conditional |
| `cancelled` | Human cancelled before terminal state | Yes |
| `integrity_violation` | SHA256 chain failure at any layer boundary | Yes |

`capability_exceeded` is terminal if the human cancels. Non-terminal if the human confirms acquisition of the missing item and the item is successfully acquired and verified in the pool.

---

### State Transitions

```
draft
  → pending_approval        (Planner generates plan, tools verified available)
  → Handanza      (Planner detects tool gap, surfaces foundation task)

pending_approval
  → approved                (human confirms through Bit)
  → Harkanza        (human denies plan)
  → draft                   (human requests revision — plan sent back to Planner)

Harkanza
  → draft                   (human revises intent and resubmits)
  → cancelled               (human explicitly cancels)

Handanza
  → draft                   (Hašatar complete — Planner re-evaluates)
  → cancelled               (human cancels foundation task or original request)

approved
  → queued                  (Job Spec accepted by Router)
  → integrity_violation     (SHA256 mismatch on Job Spec receipt)

queued
  → running                 (Router dispatches first chunk)
  → cancelled               (human cancels before execution begins)

running
  → completed               (all steps completed successfully, Planner validates)
  → partial                 (execution finished, some steps failed, Planner validates)
  → failed                  (execution failed, Planner validation declines)
  → paused                  (step requires human input — NEEDS_INFO signal)
  → capability_exceeded     (Router determines fleet cannot service at acceptable quality)
  → integrity_violation     (SHA256 mismatch on any Work Result)
  → cancelled               (human cancels during execution)

paused
  → running                 (human provides required input through Bit)
  → cancelled               (human cancels rather than providing input)

capability_exceeded
  → queued                  (human confirms acquisition of missing item, item acquired and verified)
  → cancelled               (human cancels rather than acquiring missing item)
```

### Step States

| State | Meaning | Terminal |
|---|---|---|
| `pending` | Waiting for dependencies | No |
| `running` | Currently executing | No |
| `paused` | Waiting for human input | No |
| `retrying` | Retry in progress after failure | No |
| `completed` | Finished successfully | Yes |
| `failed` | Failed — no retry remaining or retry exhausted | Yes |
| `timeout` | Exceeded time budget — treated as failed | Yes |
| `skipped` | Dependency failed, step not executed | Yes |

**Step transition rules:**
- `pending` → `running` when all declared dependencies are `completed`
- `pending` → `skipped` when any declared dependency is `failed` or `timeout` and `on_failure: abort`
- `running` → `paused` when the step emits a NEEDS_INFO signal
- `running` → `retrying` when failure occurs and retry budget remains
- A job may not reach `completed` while any step is non-terminal

### Priority Lanes

| Lane | Label | Use |
|---|---|---|
| P0 | Control plane | System integrity — node registration, capability updates, integrity violation responses |
| P1 | Interactive | Human is actively waiting for a result |
| P2 | Batch | Background work — corpus enrichment, prefill generation, scheduled analysis |
| P3 | Experimental / maintenance | Foundation generation, tool creation, zoo evaluation, low-urgency tasks |

**Lane invariants:**
- P0 preempts all other lanes — no scheduling policy may delay a P0 job
- P1 SLA is defined per deployment — the Router is responsible for meeting it
- P2 and P3 jobs fill idle capacity and are never allowed to starve P0 or P1
- Foundation generation for a gap blocking a P1 request runs at P2, not P3

## Section 3: Capability Envelope Protocol

A system's capability is advertised to the Router through structured envelopes — one per schedulable node within the system. The Foreman is the system-local owner of all envelope management, heartbeat, and health check responsibilities. The Router never probes nodes. It only knows what the Foreman tells it.

---

### Terminology

**System** — a physical device with one Foreman process, one network presence, and one `system_id`. Examples: Daemon, Ergaster, Logos, a Raspberry Pi.

**Node** — an individually schedulable compute and memory surface within a system. A system contains one or more nodes. Examples within Daemon: the CPU+system RAM pool, P100 island 1, P100 island 2, the 5060 Ti island. Each node has its own `node_id` and its own capability envelope.

**Foreman** — the system-local coordinator. Owns all node envelopes, the heartbeat, capability advertisements, and health check responses on behalf of its system. One Foreman per system.

---

### Envelope Principles

**Self-reported, not discovered.** The Router does not probe systems. The Foreman surveys its hardware, computes envelopes for each node, and advertises them. The Router trusts what it receives until behavior contradicts it.

**Capacity envelopes, not hardware identity.** A node does not say "I am an RTX 5060 Ti." It says "I have 16GB accelerator memory, 672 GB/s bandwidth, compute class dgpu, and I support fp16 and tensor_cores." No vendor name, chip model, or hardware identifier ever crosses a layer boundary.

**Per-node granularity.** Each schedulable surface within a system is a distinct node with its own envelope. The Router schedules against individual nodes. The system boundary is visible to the Router only as a grouping of nodes sharing a Foreman — expressed via `system_id` — for co-location inference.

**Soft state with mandatory renewal.** Capability declarations expire without heartbeat renewal. A silent node is an unavailable node. Reinstatement requires successful re-registration.

**Hash-verified integrity.** Every envelope carries a SHA256 hash of its canonical capability fields. The Router verifies on receipt. Mismatch triggers a reindex request.

**Tool updates propagate via hash change.** When a node pulls an updated tool from the pool, its capability profile changes, its envelope hash changes, the next heartbeat triggers `reindex_required`, and the Router receives a fresh envelope. This is the intended update propagation mechanism.

**One-way flow.** Capability advertisements flow Foreman → Router only. The Router responds with `acknowledged`, `reindex_required`, or `rejected`. It never pushes capability assumptions back.

The heartbeat is the Foreman's proof of life — one per system, carrying a per-node status summary.

**Heartbeat schema:**

```json

    {
      "node_id": "<uuid>",
      "alive": true,

      "current_load_pct": 0,
      "available_memory_gb": 0,

    }
  ]
}
```

**Heartbeat response:**

```json
{
  "schema_version": "1.0",
  "heartbeat_id": "<uuid>",
  "system_id": "<uuid>",
  "received_at": "<ISO8601>",
  "result": "<acknowledged|reindex_required|rejected>",
  "reason": "<null if acknowledged>"

```

**Protocol flow:**
1. Foreman sends heartbeat at configured interval
2. Router verifies `envelope_hash` against stored record for `system_id`
3. Hash matches → `acknowledged`
4. Hash mismatch → `reindex_required`. All nodes for this system marked `ready: false` until re-registration completes
5. No heartbeat within window → all nodes marked `alive: false, ready: false`
6. System returns after silence → full re-registration required, not a heartbeat

**Invariants:**
- The Router never reinstates a system from `alive: false` automatically
- A system that re-registers after silence is treated as a fresh registration

### Degradation Detection and Remediation

The Router trusts the envelope until behavior contradicts it.

**Degradation signals:**

| Signal | Description |
|---|---|
| Missed heartbeats | Foreman silent beyond TTL |
| Latency breach | Result latency consistently outside declared `p95` band |
| Elevated error rate | Work results returning `failed` or `timeout` at anomalous rate |
| Partial output anomaly | Results returning `partial` at rate inconsistent with workload complexity |
| Load contradiction | Heartbeat load report inconsistent with chunk assignment volume |

**Health check request:**

```json
{
  "schema_version": "1.0",
  "health_check_id": "<uuid>",
  "system_id": "<uuid>",
  "node_id": "<uuid — null if system-wide>",
  "requested_at": "<ISO8601>",
  "degradation_signals": ["<signal type>"],
  "verification_requested": ["<tool_availability|model_load_state|latency_benchmark|memory_available>"]
}
```

**Health check response:**

```json
{
  "schema_version": "1.0",
  "health_check_id": "<uuid>",
  "system_id": "<uuid>",
  "node_id": "<uuid>",
  "responded_at": "<ISO8601>",
  "status": "<healthy|degraded|failed>",
  "verification_results": {
    "tool_availability": "<verified|mismatch|unavailable>",
    "model_load_state": "<verified|mismatch|unavailable>",
    "latency_benchmark_ms": 0,
    "memory_available_gb": 0
  },
  "recommended_action": "<none|reindex|remove_from_pool|human_review_required>"
}
```

**Remediation paths:**

| Health check result | Router action |
|---|---|
| `healthy` | Resume normal scheduling. Log false positive. |
| `degraded` + `reindex` | Mark node `ready: false`. Request re-registration. Resume on success. |
| `degraded` + `remove_from_pool` | Mark node `alive: false`. Surface to Bit as node health alert. |
| `failed` | Mark node `alive: false`. Surface to Bit as critical alert. |
| `human_review_required` | Mark node `ready: false`. Surface full report to Bit. Human decides. |
| No response within TTL | Treat as `failed`. |

### Tool Pool and Gap Detection

The gap-as-project mechanism is shared between node-detected capability gaps and Planner-detected tool gaps for specific task requests. Both surface in Bit's dashboard, require human confirmation, and route through the same P2/P3 queue.

---

### Tool Population Rules

- **Pull, don't push** — nodes pull tools from pool, pool does not push to nodes
- **Compatibility first** — tool installed only if node satisfies all compatibility requirements
- **Dependency resolution** — all dependencies must be installable; if not, both dependency and dependent tool recorded as gaps
- **Minimum memory enforcement** — tools with `min_memory_gb` exceeding available memory not installed, recorded as gap
- **No partial installation** — tool either installs completely or not at all
- **Tool hash verification** — every artifact verified against `tool_hash` before installation completes

---

### Tool Updates and Envelope Propagation

When an updated tool version is available:
1. Foreman detects version delta on next pool sync
2. Foreman pulls and installs new version
3. Installed tool version in node envelope changes
4. Envelope hash changes
5. Next heartbeat triggers `reindex_required`
6. Foreman submits full re-registration with updated envelope
7. Router updates registry, scheduling resumes

**Rollback:** if updated tool fails validation, Foreman retains previous version, records failure, surfaces to Bit as tool update failure alert.

---

### Gap Lifecycle

```
detected (during onboarding survey)
  → reported (included in system registration payload)
    → surfaced (Router passes to Planner, Planner surfaces to Bit dashboard)
      → confirmed (human selects action)
        → queued (Hašatar task created at P2/P3)
          → in_progress
            → resolved (new tool added to pool, node re-registers)
            → failed (surfaces back to Bit as project failure)
      → deferred (human acknowledges but does not act)
      → dismissed (human determines gap is acceptable)
```

Gap resolution for one system automatically benefits all systems with the same gap on their next pool sync.

## Section 5: Task Package Schema

A Task Package is a self-contained, versioned execution module. It is the unit of registered capability in the Concierge system. The Planner matches intent against Task Packages. The Router executes them. The human approves them.

The kernel module analogy is precise: a Task Package declares its interface, its dependencies, its resource requirements, and its behavior contract. The system loads it, verifies it, and executes it within defined bounds.

---

### Package Principles

**Spec before code.** A Task Package is written before any implementation that serves it.

**Immutable once registered.** Changes produce a new version. The old version remains available for in-flight jobs.

**Pre-acknowledged action scope.** Each package declares the class of actions it performs. This is the pre-acknowledgement — the human has implicitly approved this action class by installing the package. Execution of a specific instance still requires explicit confirmation.

**Self-describing confidence matching.** Each package declares how the Planner should score intent against it.

**`high_stakes`:** when true on a pipeline step, the Router escalates output verification for that step to the High Table verifier tier regardless of ensemble agreement level. **`n_candidates` and `candidate_selection`:** some packages produce multiple candidate outputs by design. `candidate_selection: human` surfaces all candidates to Bit's gallery viewer for human selection. `auto_best` selects by internal scoring. `auto_all` returns all candidates as artifacts.

**`recommended_models` and `quality_impact_if_absent`:** the mechanism for the expanded `capability_exceeded` trigger. If a recommended model with `quality_impact_if_absent: critical` or `significant` is absent from the pool, the Router surfaces a `capability_exceeded` condition.

**`nidaba_enrichment`:** when true, the package's outputs and execution context are fed to NIDABA as a side effect of completion. This is how the corpus grows from system activity.

---

### Package Versioning

When a new version is registered, the old version remains available for in-flight jobs. When no active jobs reference an old version, it is archived — retained for audit but not matchable. The `supersedes` field creates an auditable chain of package evolution.

---

### Task Package Creation Wizard

**Stage 1 — Intent clarification.** Planner surfaces clarifying questions about purpose, inputs, outputs, and action scope.

**Stage 2 — Draft generation.** Planner generates a draft Task Package. Required fields without clear answers flagged for human completion.

**Stage 3 — Human review.** Draft presented to human in Bit. Human may edit any field or approve as-is.

**Stage 4 — Validation.** Draft validated against schema. Errors surfaced with plain-language explanations.

**Stage 5 — Confirmation.** Human explicitly confirms registration — separate from review.

**Stage 6 — Registration.** Package added to registry with new `package_id`, `package_version: 1.0.0`, computed `package_hash`. Immediately available for intent matching.

Interrupted Wizard sessions are saved as draft packages — visible in Bit's dashboard but not registered until Stage 6 completes.

---

### Invariants

- A registered package is immutable — changes produce a new version
- `package_hash` covers all canonical package fields and is verified on registry load
- `requires_explicit_confirmation_per_run` is always true
- A package that declares `required_tools` not in the pool cannot be executed — Planner's tool availability query catches this
- A package with `recommended_models` at `quality_impact_if_absent: critical` and those models absent triggers `capability_exceeded`
- The Wizard produces a draft — not registered until explicit human confirmation

### Artifact Lifecycle

```
produced (Workbee writes artifact, step emits artifact.produced event)
  → stored (written to workspace artifact store, hash computed)
    → validated (acceptance criteria checked)
      → passed
          → delivered (Planner passes artifact reference to Bit)
            → viewed (human opens in Bit viewer)
            → archived (moved to long-term storage per retention policy)
      → failed
          → surfaced (Planner surfaces failure to Bit with detail)
            → accepted despite failure (human overrides — logged)
            → rejected (artifact discarded, step marked failed)
```

**Artifact transmission:** artifacts produced on remote nodes are transmitted to the workspace artifact store. Transmission verified by comparing hash at node against hash after receipt. Mismatch is an integrity violation.

**Candidate artifacts:** for packages producing N candidates, all candidates are stored as separate ArtifactRecords. When `candidate_selection: human`, all are delivered to Bit's gallery viewer simultaneously. Non-selected candidates are retained — not deleted.

---

### Planner Result Validation Sequence

1. **Integrity check** — verify SHA256 chain. Any mismatch → `integrity_violation`.
2. **Completeness check** — verify all required steps completed.
3. **Acceptance criteria evaluation** — evaluate each artifact against Task Package criteria.
4. **Quality gate evaluation** — `abort` gates that fail cause failure; `warn` gates log and continue; `surface_to_human` gates pause delivery.
5. **Decision** — accept or decline, log everything, pass up or down as appropriate.

}

## Section 7: Router Scheduling Logic

The Router is the operational core of the Concierge system. It is the only component that knows the full state of the fleet. Its internal design — scheduling algorithm, chunk sizing logic, node selection strategy, queue management, and concurrency model — is the subject of a dedicated design session with multi-model peer review. That session will produce a Router Design Specification as a separate document. What is specified here is the Router's contract boundary only.

---

### Router Responsibilities

The Router owns the following and nothing else:

- Maintaining the node registry
- Responding to Planner tool availability queries
- Receiving approved Job Specs from the Planner
- Decomposing execution plans into Work Chunks
- Managing the work pool
- Matching chunks to capable nodes
- Tracking chunk lease state
- Aggregating Work Results into ExecutionState
- Returning completed ExecutionState to the Planner
- Emitting scheduling events to the event stream
- Enforcing priority lane scheduling
- Initiating health checks on degraded nodes
- Surfacing `capability_exceeded` conditions to the Planner

The Router does not validate business logic, evaluate acceptance criteria, communicate with Bit, or know what the human asked for.

---

### Router Contract Boundary — Inputs

| Input | From | Schema defined in |
|---|---|---|
| System registration | Foreman | Section 3 |
| Heartbeat | Foreman | Section 3 |
| Health check response | Foreman | Section 3 |
| Tool availability query | Planner | Section 1 |
| Approved Job Spec | Planner | Section 1 |
| Work Result | Foreman | Section 3 |
| Node removal instruction | Human via Bit/Planner | Section 3 |

### Router Contract Boundary — Outputs

| Output | To | Schema defined in |
|---|---|---|
| Registration response | Foreman | Section 3 |
| Heartbeat response | Foreman | Section 3 |
| Health check request | Foreman | Section 3 |
| Tool availability response | Planner | Section 1 |
| Work Chunk | Foreman (via work pool) | Section 1 |
| ExecutionState (completed) | Planner | Section 6 |
| Scheduling events | Event stream | Section 6 |
| Capability exceeded signal | Planner | Section 2 |

---

### Priority Lane Enforcement

| Lane | Label | Scheduling behavior |
|---|---|---|
| P0 | Control plane | Preempts all. No delay permitted. |
| P1 | Interactive | Must meet P1 SLA defined in deployment configuration. Pathfinder WoL signals triggered at P1 classification before job reaches Router. |
| P2 | Batch | Fills idle capacity. Never starves P0 or P1. |
| P3 | Experimental / maintenance | Lowest priority. Fills capacity not consumed by P0–P2. |

**MDSD backfill:** during SLA interleave windows when P1 queue is empty and P0 has no pending work, Router fills idle node capacity with P2 and P3 work. No cycle is wasted.

---

### Work Pool Model

The work pool is the Router's internal queue of chunks available for Foremen to pull.

**Pull model invariants:**
- No chunk names a specific node as its target
- Foreman pulls only chunks whose requirements fall within its nodes' declared envelopes
- Chunks are leased for `lease_ttl_ms` — expired leases return to pool
- Chunk states: `available`, `leased`, `completed`, `failed`, `expired`

---

### Chunk Sizing Requirements

Invariant requirements regardless of algorithm chosen:
- A chunk must be executable by a single node
- A chunk's compute requirements must fit within a single node's declared envelope
- A chunk may not span multiple steps
- Chunk size must respect node's declared `max_context_tokens` for inference steps
- Chunk size must respect node's declared `parallel_chunk_capacity`

---

### Node Matching Requirements

Invariant requirements regardless of algorithm chosen:
- `compute_class` in chunk must be satisfied by node's `node_class`
- `memory_requirement_gb` must not exceed node's `available_memory_gb`
- All `accelerator_features` in chunk must be present in node's `compute.features`
- Node must have all `required_tools` for the step installed
- Node must be `alive: true` and `ready: true` at time of assignment
- No assignment to nodes with active `degradation_flags`

---

### Temporary Sharding

**Shard eligibility:**
- Step action must be declared shard-compatible in Task Package
- Multiple nodes with sufficient combined capacity must be available
- P1 work: intra-system sharding only (`latency_class: intra_system`)
- P2/P3 work: cross-system sharding permitted

**Shard attempt sequence:**
1. Router identifies shard candidate pool from registry
2. Router computes whether combined envelope satisfies step requirements
3. If satisfied: create coordinated chunk set, assign to shard nodes
4. If not satisfied: return `capability_exceeded` to Planner

Shard results aggregated by Router before returning to Planner. Sharding is invisible above the Router boundary.

---

### High Table Escalation

**Escalation triggers:**
- Ensemble outputs below quorum agreement threshold
- Steps flagged `high_stakes: true` in Task Package
- Steps in packages with quality gates requiring model-scored validation
- `capability_exceeded` conditions where a recommended verifier model is available

Escalation is transparent above the Router boundary — Planner and Bit see a step result, not that it was verified by the High Table.

---

### capability_exceeded Signal

**Trigger conditions:**
- No node satisfies step compute requirements after sharding attempted
- Recommended model with `quality_impact_if_absent: critical` or `significant` not in pool
- Required tool not available on any ready node
- Recommended tool or transformer with significant quality impact absent

**capability_exceeded payload:**

```json
{
  "schema_version": "1.0",
  "signal_id": "<uuid>",
  "job_id": "<uuid>",
  "step_id": "<step_id that triggered the condition>",
  "execution_id": "<uuid>",
  "issued_at": "<ISO8601>",
  "trigger_type": "<compute_ceiling|missing_model|missing_tool|missing_transformer>",
  "missing_items": [
    {
      "item_id": "<model_id or tool_id>",
      "item_type": "<model|tool|transformer>",
      "role": "<primary|verifier|drafter|classifier>",
      "quality_impact": "<critical|significant|minor>",
      "resolution": "<download_existing|build_new_tool|acquire_hardware>",
      "resolution_detail": "<human-readable description>",
      "blocking": true
    }
  ],
  "sharding_attempted": false,
  "fleet_ceiling": {
    "max_single_node_memory_gb": 0,
    "max_shard_memory_gb": 0,
    "available_compute_classes": ["<cpu|igpu|dgpu|npu|ane>"]
  },
  "degraded_alternative_available": false,
  "degraded_alternative_description": "<null if none>"
}
```

---

### Router Invariants

- Zero knowledge of hardware identity — sees capability envelopes only
- Never pushes work to nodes — all chunk acquisition is pull-based
- Never modifies a chunk after posting to work pool
- Never modifies a Job Spec after receipt
- Never communicates with Bit directly
- Never evaluates business logic or acceptance criteria
- Always returns `capability_exceeded` signal before marking a job failed — human decides
- Priority lane assignments set by Planner, enforced by Router — Router does not change them
- Shard results aggregated before returning to Planner — sharding invisible above boundary
- Router Design Specification governs internal implementation — this section governs contract boundary only

## Section 8: SHA256 Integrity Chain

Every artifact that crosses a layer boundary carries a SHA256 hash of its canonical fields, computed at emission and verified at receipt. A mismatch at any point is a hard failure. No silent corruption propagates through the stack.

The integrity chain is a correctness mechanism against accidental corruption, transmission error, serialization inconsistency, and implementation bugs.

---

### Chain Structure

```
Intent Artifact
  sha256_intent = SHA256(intent_id + issued_at + intents[] + questions[] + status)

Execution Plan
  sha256_plan = SHA256(plan_id + intent_id + sha256_intent + package_id + package_version + approximated + steps[])

Approved Job Spec
  sha256_job_spec = SHA256(job_id + plan_id + sha256_plan + approved_at + approver + approval_note)

Work Chunk
  sha256_chunk = SHA256(chunk_id + job_id + plan_id + step_id + sha256_job_spec + action + inputs + compute_requirements)

Work Result
  sha256_result = SHA256(result_id + chunk_id + job_id + step_id + sha256_chunk + status + output)

ExecutionState (final)
  sha256_execution = SHA256(execution_id + job_id + plan_id + sha256_job_spec + started_at)
```

---

### Canonical Representation

**Rules:**
- All fields serialized as UTF-8 JSON
- Object keys sorted alphabetically at all nesting levels
- Arrays serialized in declared order — array order is semantically significant, must not be sorted
- Null fields included with explicit `null` — not omitted
- Timestamps: ISO8601, UTC, millisecond precision — `2026-03-28T22:15:30.000Z`
- UUID fields: lowercase hyphenated

**Excluded from hashing:**

| Artifact | Excluded fields |
|---|---|
| Intent Artifact | `queued_at`, `connection_state` |
| Execution Plan | `estimated_duration_ms` |
| Work Chunk | `lease_ttl_ms` |
| Work Result | `duration_ms`, `retries_used` |
| ExecutionState | `status`, `progress`, `completed_at`, `planner_validation` |

---

### Verification Points

| Artifact | Emitted by | Verified by | On mismatch |
|---|---|---|---|
| Intent Artifact | Bit | Planner | Reject artifact, return error to Bit, log violation |
| Execution Plan | Planner | Bit (at approval) | Reject plan, return to Planner, log violation |
| Approved Job Spec | Planner | Router | Reject job, mark `integrity_violation`, log violation |
| Work Chunk | Router | Foreman | Reject chunk, return to pool, log violation |
| Work Result | Foreman | Router | Reject result, treat step as failed, log violation |
| ExecutionState | Router | Planner | Reject execution, mark `integrity_violation`, log violation |

**Bit verifies the Execution Plan at approval time.** The approval signal carries the verified plan hash:

```json
{
  "schema_version": "1.0",
  "approval_id": "<uuid>",
  "plan_id": "<uuid>",
  "job_id": "<uuid>",
  "approved_at": "<ISO8601>",
  "approver": "<human identifier>",
  "verified_plan_hash": "<sha256_plan — Bit recomputed before displaying to human>",
  "approval_note": "<optional>"
}
```

The Router verifies `verified_plan_hash` matches the plan hash it holds before creating the Job Spec.

---

### integrity_violation Job State

On mismatch detection:
1. Detecting layer rejects artifact, emits `integrity_violation` event
2. All in-flight chunks for the job are cancelled
3. Job marked `integrity_violation` in ExecutionState
4. Planner surfaces violation to Bit with plain-language explanation
5. Violation logged with full detail — which artifact, which layer, expected hash, received hash, timestamp

Integrity violation events are never purged. Retained indefinitely regardless of retention policy.

---

### Integrity Event Schema

```json
{
  "schema_version": "1.0",
  "violation_id": "<uuid>",
  "job_id": "<uuid>",
  "execution_id": "<uuid — null if violation before execution started>",
  "detected_at": "<ISO8601>",
  "detecting_layer": "<bit|planner|router|foreman>",
  "artifact_type": "<intent_artifact|execution_plan|job_spec|work_chunk|work_result|execution_state>",
  "artifact_id": "<uuid of artifact that failed verification>",
  "expected_hash": "<sha256 the detecting layer computed>",
  "received_hash": "<sha256 declared in the artifact>",
  "disposition": "<artifact_rejected|job_terminated|chunk_cancelled>"
}
```

---

### Hash Algorithm and Implementation Notes

**Algorithm:** SHA256 only. Outputs are lowercase hex strings — 64 characters.

**Canonical JSON:** key ordering must be applied recursively — nested objects must also have sorted keys. Null fields must be included, not omitted. Most JSON libraries do not sort keys by default — a canonical serialization utility must be implemented and tested explicitly.

**Timestamp precision:** `2026-03-28T22:15:30Z` and `2026-03-28T22:15:30.000Z` produce different hashes. Millisecond precision with `.000` suffix required even for round-second timestamps.

**Cross-language consistency:** hashes computed by a Python Planner and verified by a Go Router must agree. Cross-language canonical serialization tests are a required part of the test suite — not optional.

**Hash computation timing:** hashes are computed immediately before transmission and verified immediately after receipt.

---

### Integrity Chain Invariants

- SHA256 is the only hash algorithm — no substitution or fallback
- Every artifact crossing a layer boundary carries a hash
- Every receiving layer verifies the hash before processing
- A hash mismatch is always a hard failure — no soft mismatch handling
- `integrity_violation` is a terminal job state — no recovery, no retry
- Integrity violation events retained indefinitely
- Human always notified of integrity violation — never silently discarded
- Cross-language canonical serialization consistency is a test suite requirement
- Approval signal carries verified plan hash — Router verifies before creating Job Spec

## Section 9: Memory System Interface Contract

The Memory System is a distinct subsystem of Concierge with its own design specification — `Concierge_Memory_Spec_v5.md`. This section defines only the interface contract: the tier identifiers, what each tier contains, and the read/write rules each layer must respect. Internal memory system design is deferred to the Memory Spec.

---

### Memory Tiers

| Tier | Name | Contents | Persistence |
|---|---|---|---|
| Tier 0 | Active context | Current session state — in-flight intent, active job refs, current conversation | Session lifetime only — not persisted across sessions |
| Tier 1 | Event log | Append-only execution log — all job events, state transitions, integrity events | Permanent — never purged |
| Tier 2 | Structured facts | User preferences, settings, named entities, calibrated behavioral patterns | Persistent — explicit writes only |
| Tier 3 | Vector archive | Semantic search index over corpus, execution history, enriched knowledge | Persistent — grown by NIDABA enrichment and corpus write operations |

These tier definitions are current as of v5. The Memory Spec may introduce sub-tiers or additional tiers as the memory system design matures.

---

### Layer Read/Write Rules

| Layer | Tier 0 | Tier 1 | Tier 2 | Tier 3 |
|---|---|---|---|---|
| Bit | Read/Write | Read | Read | Read |
| Planner | Read | Read/Write | Read | Read/Write (NIDABA enrichment only) |
| Router | None | Write (scheduling events) | None | None |
| Foreman | None | Write (execution events) | None | None |
| Workbee | None | Write (step events) | None | None |

**Write restrictions:**
- Tier 0: Bit's working memory only — Planner updates via Bit's defined interface, not directly
- Tier 1: append-only — no layer may modify or delete a Tier 1 record once written
- Tier 2: explicit human-confirmed operations only — no silent updates to user preferences or settings
- Tier 3: NIDABA enrichment only — a declared package property, not an implicit layer capability

---

### Memory References in Artifacts

The `context_refs` field in the Intent Artifact carries references to memory tier entries that provide context for the intent. The Planner resolves these references before plan generation.

```json
"context_refs": [
  {
    "tier": 2,
    "ref_id": "<identifier of the specific memory entry>",
    "ref_type": "<preference|setting|named_entity|behavioral_pattern>"
  }
]
```

The Memory Spec defines the full reference resolution protocol.

---

### Memory System Invariants

- No layer reads from or writes to a tier not listed in its row in the read/write table
- Tier 1 is append-only across all layers — immutable once written
- Tier 2 writes require human confirmation — no layer may silently update user preferences
- Tier 3 writes occur only as a declared side effect of Task Package execution with `nidaba_enrichment: true`
- Memory tier definitions may expand in the Memory Spec — this interface contract remains the authoritative boundary definition for the Technical Spec
